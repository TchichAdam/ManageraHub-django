\chapter{Réalisation et Interfaces}

\section{Introduction}
Ce chapitre détaille la phase d'implémentation technique de ManageraHub. Il décrit l'environnement de développement matériel et logiciel, détaille l'architecture MVT de Django appliquée aux différents modules, justifie les optimisations de requêtes implémentées, présente les écrans finaux de l'application via des conteneurs de captures d'écran, et analyse les difficultés rencontrées lors de la phase d'intégration responsive.

\section{Environnement de travail}
\subsection{Environnement matériel}
Le développement de la plateforme a été réalisé sur une machine portable équipée d'un processeur Intel Core i7 cadencé à 2.8 GHz, de 16 Go de mémoire vive (RAM) et d'un stockage SSD de 512 Go. Cette configuration a permis de faire tourner le serveur de développement local et la base de données relationnelle de manière fluide.

    \item \textbf{Langage et Framework :} Python (version 3.9) et Django (version 4.2 LTS).
    \item \textbf{IDE / Éditeurs :} Visual Studio Code pour l'écriture et le débogage du code Python\allowbreak/Django et des fichiers statiques.

\subsection{Environnement de déploiement cloud (Production)}
Pour la mise en production de ManageraHub, nous avons opté pour une infrastructure cloud performante et évolutive :
\begin{itemize}
    \item \textbf{Fournisseur Cloud :} Oracle Cloud Infrastructure (OCI).
    \item \textbf{Machine Virtuelle (VM) :} Instance de calcul Compute sous Ubuntu Server 22.04 LTS (IP publique : 51.170.143.223).
    \item \textbf{Serveur Web \& Reverse Proxy :} Nginx configuré pour écouter sur les ports standards HTTP/HTTPS et rediriger le trafic vers le serveur d'application interne.
    \item \textbf{Serveur d'application WSGI :} Gunicorn gérant les processus de l'application Django en arrière-plan (port local 8000).
    \item \textbf{Résolution DNS :} Utilisation du service de wildcard DNS de confiance \texttt{sslip.io} permettant d'attribuer le nom de domaine résolvable \texttt{51-170-143-223.sslip.io} pointant directement vers l'adresse IP publique de la VM.
    \item \textbf{Base de données :} SQLite (\texttt{db.sqlite3}), choisi pour sa simplicité et sa légèreté lors des phases de validation.
\end{itemize}

\section{Architecture technique du code source}
Le code de ManageraHub est structuré au sein d'une seule application Django principale nommée \texttt{app1} (sous le projet global \texttt{monprojet}). Cette approche centralisée simplifie grandement les imports de modèles et l'architecture globale.

\subsection{Modèles (app1/models.py)}
Voici un exemple d'extrait du fichier de définition de nos modèles de profils CandidateProfile et CompanyProfile, liés au modèle utilisateur par défaut de Django (\texttt{User}) :

from django.db import models
from django.conf import settings

def candidate_profile_file_path(instance, filename):
    user_id = getattr(instance, "user_id", None)
    if user_id is None and hasattr(instance, "profile_id"):
        user_id = instance.profile.user_id
    return f"candidate_profiles/user_{user_id}/{filename}"

def company_profile_file_path(instance, filename):
    return f"company_profiles/user_{instance.user_id}/{filename}"

class CandidateProfile(models.Model):
    user = models.OneToOneField(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name="candidate_profile",

    phone_number = models.CharField(max_length=40, blank=True)
    address = models.CharField(max_length=255, blank=True)

    education_level = models.CharField(max_length=120, blank=True)
    skills = models.TextField(blank=True)
    experience_summary = models.TextField(blank=True)
    headline = models.CharField(max_length=180, blank=True)
    bio = models.TextField(blank=True)
    profile_picture = models.FileField(upload_to=candidate_profile_file_path, blank=True)
    cv_file = models.FileField(upload_to=candidate_profile_file_path, blank=True)
    cover_letter_file = models.FileField(upload_to=candidate_profile_file_path, blank=True)
    is_public = models.BooleanField(default=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

class CompanyProfile(models.Model):
    user = models.OneToOneField(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name="company_profile",
    )
    company_name = models.CharField(max_length=180, blank=True)

    city = models.CharField(max_length=120, blank=True)

    website = models.URLField(max_length=300, blank=True)
    description = models.TextField(blank=True)
    logo = models.ImageField(upload_to=company_profile_file_path, blank=True)
    background_image = models.ImageField(upload_to=company_profile_file_path, blank=True)

    updated_at = models.DateTimeField(auto_now=True)

\subsection{Logique métier (views.py) et optimisation des requêtes BDD}
Afin d'éviter le problème classique des requêtes N+1 (N+1 Query Problem), nous avons optimisé le chargement des publications dans le fil d'actualités social. En utilisant \texttt{select\_}\allowbreak\texttt{related} pour précharger l'auteur du post et l'entreprise taguée, et \texttt{prefetch\_}\allowbreak\texttt{related} pour les commentaires, leurs auteurs et les mentions d'utilisateurs, nous réduisons le nombre de requêtes SQL de plus de 80\% sur les pages de flux social.

\begin{lstlisting}[language=Python, caption=Optimisation des requêtes du Feed social dans views.py]
from django.shortcuts import render
from .models import Post

def candidate_feed_view(request):
    # Optimisation par prechargement select_related et prefetch_related
    posts = Post.objects.select_related("author", "tagged_company").prefetch_related(
        "likes", "comments", "comments__author", "tagged_users", "tagged_users__candidate_profile"
    ).all()
    
    return render(request, 'candidate/feed.html', {'posts': posts})
\end{lstlisting}

\subsection{Authentification Sociale (OAuth 2.0 via django-allauth)}
Pour offrir une expérience utilisateur moderne et fluide, nous avons intégré l'authentification sociale à l'aide de la bibliothèque \texttt{django-allauth}. Les candidats peuvent ainsi s'inscrire ou se connecter en un clic avec leurs comptes Google ou GitHub.

La configuration associe les comptes sociaux récupérés à des profils \texttt{User} locaux. Si l'adresse e-mail retournée par le fournisseur OAuth n'est pas encore enregistrée, un compte est créé à la volée. Nous avons personnalisé les gabarits (\texttt{socialaccount/}\allowbreak\texttt{signup.html} et \texttt{socialaccount/}\allowbreak\texttt{login\_error.html}) pour maintenir la charte graphique de la plateforme (Dark Plum et Periwinkle) et afficher des messages d'erreur clairs en cas d'échec d'autorisation réseau (comme par exemple l'erreur d'inscription sans CSS).

\subsection{Système de notifications par e-mail transactionnel (SMTP)}
Le système d'e-mail de ManageraHub a été entièrement refondu pour envoyer des e-mails transactionnels HTML modernes et professionnels. Les notifications par e-mail sont générées dynamiquement lors d'événements clés :
\begin{itemize}
    \item \textbf{Bienvenue candidat/\allowbreak entreprise :} Message personnalisé envoyé dès la création du compte.
    \item \textbf{Validation d'entreprise :} Notification de l'approbation du compte par l'administrateur.
    \item \textbf{Rejet d'entreprise :} Notification du refus du compte avec le motif saisi.
    \item \textbf{Changement de statut de candidature :} Notification en temps réel du candidat (ex: accepté, rejeté, entretien) avec des cartes de statut colorées pour un impact visuel optimal.
\end{itemize}

L'implémentation repose sur le protocole SMTP de Gmail avec un mot de passe d'application. Nous utilisons \texttt{EmailMultiAlternatives} dans \texttt{app1/emails.py} pour joindre une version texte brut en alternative au gabarit HTML.

Pour garantir la cliquabilité et le bon rendu visuel des boutons de redirection (comme le bouton d'accès au profil ou à l'offre) sur tous les clients mail (Gmail, Outlook, Apple Mail), nous avons mis en œuvre le pattern de conception \textbf{Bulletproof Button}. Ce pattern combine l'inline-block HTML/CSS pour les clients standard et le balisage VML (Vector Markup Language) spécifique pour assurer la compatibilité Microsoft Outlook.

Le code suivant montre l'implémentation de la fonction d'envoi d'e-mails HTML :

\begin{lstlisting}[language=Python, caption=Fonction d'envoi d'e-mails HTML personnalisés dans app1/emails.py]
from django.core.mail import EmailMultiAlternatives
from django.template.loader import render_to_string
from django.utils.html import strip_tags
from django.conf import settings

def send_html_email(subject, template_name, context, recipient_list, from_email=None):
    # Injection automatique de la variable de l'URL du site
    context['site_url'] = getattr(settings, 'SITE_URL', 'http://127.0.0.1:8000')
    
    # Rendu du template HTML
    html_content = render_to_string(template_name, context)
    text_content = strip_tags(html_content)
    
    email = EmailMultiAlternatives(
        subject=subject,
        body=text_content,
        from_email=from_email or settings.DEFAULT_FROM_EMAIL,
        to=recipient_list,
    )
    email.attach_alternative(html_content, "text/html")
    try:
        return email.send(fail_silently=False)
    except Exception as e:
        return False
\end{lstlisting}

\section{Sécurité applicative mise en œuvre}
Le framework Django intègre nativement des protections contre les vulnérabilités courantes du web. Nous avons configuré et vérifié ces mécanismes :
\begin{itemize}
    \item \textbf{Protection CSRF (Cross-Site Request Forgery) :} Tous les formulaires POST génèrent un jeton unique vérifié côté serveur.
    \item \textbf{Protection XSS (Cross-Site Scripting) :} Le moteur de rendu de gabarit Django échappe automatiquement les balises HTML et JavaScript non autorisées.
    \item \textbf{Hashage des mots de passe :} Utilisation du hashage itératif PBKDF2 avec sel dynamique pour le stockage des identifiants utilisateurs.
\end{itemize}

\section{Difficultés rencontrées et solutions apportées}
Durant la phase finale d'intégration de la charte graphique et de mise en œuvre du responsive design, plusieurs défis techniques ont dû être surmontés :
\begin{enumerate}
    \item \textbf{Alignement et centrage des formulaires et des tableaux de bord :} Sur les terminaux mobiles et tablettes, certains formulaires d'authentification et de profilage apparaissaient décentrés ou présentaient des problèmes de débordement horizontal. Nous avons résolu cela en restructurant nos feuilles de style (\texttt{static/css/main.css}) pour utiliser un système de grille CSS (CSS Grid) et de boîte flexible (Flexbox) avec des marges auto-calculées, garantissant un centrage géométrique parfait de tous les boutons et entrées sur mobile.
    \item \textbf{Gestion des documents légaux :} L'ICE de l'entreprise doit obligatoirement faire 15 chiffres. Nous avons développé des validateurs personnalisés côté modèle Django (en utilisant les fonctions natives \texttt{isdigit()} et \texttt{len()}) pour bloquer l'enregistrement de valeurs invalides directement lors de la soumission du formulaire d'inscription entreprise.
    \item \textbf{Fichiers médias :} Les CV, photos de profil et lettres de motivation contiennent des informations personnelles. Pour éviter qu'ils ne soient mélangés, nous avons implémenté des fonctions de génération de chemins de stockage dynamiques et compartimentés (comme les fonctions de stockage de profil ou de candidature) qui organisent automatiquement les téléversements dans des répertoires sécurisés par identifiants d'utilisateurs et d'offres.
    \item \textbf{Mise à jour d'état en temps réel (Tracker) :} Interroger intensivement le serveur pour actualiser la timeline de candidature peut surcharger la base de données. Nous avons résolu ce problème en développant un point d'API Django dédié et optimisé qui renvoie uniquement les données JSON nécessaires, couplé à une scrutation (polling) asynchrone côté client. De plus, l'overlay de scan radar ne s'active visuellement que lorsqu'une réelle transition d'état est détectée en base de données, alliant fluidité et économie de requêtes.
    \item \textbf{Expérience fluide du fil d'actualité (Social Feed) :} Le chargement d'un grand nombre de publications avec images et vidéos ralentissait l'affichage global sur mobile. Nous avons implémenté une pagination dynamique par défilement infini (Infinite Scroll) en JavaScript. Pour combler le temps de chargement asynchrone, nous avons conçu des squelettes de chargement (skeletons loaders) animés en CSS, masquant le délai réseau et rendant l'interface fluide.
    \item \textbf{Surcharge graphique d'OAuth et django-allauth :} L'intégration de la connexion sociale Google et GitHub a initialement mené à des pages de gestion de compte et de confirmation d'inscription sans styles CSS, dégradant l'expérience. Nous avons résolu cette difficulté en surchargeant systématiquement tous les templates de \texttt{django-allauth} (notamment \texttt{signup.html} et \texttt{login\_error.html}) en y intégrant la structure HTML sémantique et la charte Dark Plum personnalisée de la plateforme.
    \item \textbf{Rendu des e-mails HTML et cliquabilité des boutons (SMTP) :} L'envoi d'e-mails HTML modernes contenant des boutons de redirection a révélé des problèmes d'affichage et de cliquabilité sur des clients de messagerie restrictifs (comme Microsoft Outlook ou Gmail mobile), ainsi qu'un classement fréquent des messages en spam. Nous avons surmonté cela en configurant le protocole SMTP Gmail avec des mots de passe d'application sécurisés pour authentifier le domaine d'envoi, et en concevant les boutons avec le pattern hybride \textbf{Bulletproof Button} (combinant VML pour Outlook et balises HTML structurelles \texttt{display: inline-block} avec styles CSS inline pour le reste des clients).
    \item \textbf{Redirection de callbacks OAuth sans nom de domaine enregistré :} Les API d'authentification Google et GitHub exigent des domaines publics résolvables pour configurer les URL de retour (Redirect URIs) et interdisent l'utilisation d'adresses IP brutes de serveurs de production. Sans nom de domaine enregistré, nous avons configuré et exploité la solution wildcard DNS gratuite \texttt{sslip.io} pour associer notre adresse IP publique de VM Oracle Cloud (\texttt{51.170.143.223}) au nom de domaine résolvable \texttt{51-170-143-223.sslip.io}, permettant de mener à bien les tests d'authentification sociale en conditions réelles de production.
\end{enumerate}
